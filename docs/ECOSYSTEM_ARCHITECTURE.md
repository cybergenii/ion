# Package ecosystem architecture (roadmap)

This document describes what is needed to grow beyond a handful of packages toward a **large, trustworthy catalog** (for example 200+ packages), as referenced in the [README roadmap](../README.md#phase-5-production-months-13-18). It complements the **client** implementation in this repository (`src/registry/ion.rs` and related code).

## What exists today (Ion CLI)

The Ion registry is consumed as a **sparse index** over HTTPS:

- `GET https://registry.ion-cpp.dev/index/{name}.json` returns an array of version entries.
- Each entry includes: `version`, `tarball` URL, `checksum` (SHA-256), optional `description`, `homepage`, `license`, `cmake_targets`, `dependencies`, `features`, and `yanked`.

The CLI resolves semver ranges, downloads tarballs, verifies checksums, caches artifacts, and generates CMake wiring. **Publishing** (uploading new versions, yanking, namespaces) is **not** implemented in this repo yet; that belongs to a **registry service** and **maintainer workflows** described below.

## Goals for a scaled ecosystem

| Goal | Meaning |
|------|--------|
| **Discoverability** | Search, categories, and consistent naming so users find packages quickly. |
| **Reproducibility** | Lockfiles + immutable version entries + checksums (already aligned with `ion.lock`). |
| **Trust** | Clear ownership, license metadata, optional signing, and abuse handling. |
| **Maintainability** | Automated validation so maintainers do not hand-edit JSON for every release. |

## Proposed components

### 1. Package artifact contract

Every published version should ship as a **source tarball** with a predictable layout:

- `ion.toml` at the repository root (same schema as user projects).
- Buildable with `ion build` / CMake as generated by Ion for dependents.
- Declared **dependencies** using Ion registry coordinates (or documented fallbacks).
- **CMake export surface** documented via `cmake_targets` in the index so downstream `CMakeLists.txt` generation stays reliable.

Optional later: **prebuilt binaries** per platform; that adds artifact matrices, ABI stability rules, and storage costs—usually a second phase after source-only works at scale.

### 2. Registry service (server-side, separate from this CLI repo)

Not all of this must ship on day one; it is the natural shape of the system as the catalog grows.

| Concern | Direction |
|---------|-----------|
| **Index storage** | Static JSON files on a CDN (current direction) and/or a small API that serves the same schema. |
| **Blob storage** | Object storage (e.g. S3-compatible) for tarballs; immutable URLs per `(name, version)`. |
| **Publish path** | Authenticated API or trusted CI integration: upload tarball + metadata → validate → append index entry. |
| **Yank / deprecation** | Mark `yanked: true` without deleting blobs; clients already respect yanked in `IonRegistry`. |
| **Mirrors** | Same index schema; client already supports `mirrors` in `~/.config/ion/config.toml`. |

Implementing the service can live in **another repository** (e.g. `ion-registry`) so this repo stays focused on the CLI and analysis tooling.

### 3. Build and release pipeline (per package)

To reach hundreds of packages without manual errors, each library needs a **repeatable pipeline**:

1. **Tag** a semver release in Git.
2. **CI** runs on pull requests and on `main`: `cargo`-style steps translated to Ion—e.g. `ion install`, `ion build`, `ion test`, `ion check` (and optionally cross-compilation smoke tests).
3. **Package** a clean tarball (no `build/`, no secrets) matching the index entry.
4. **Publish** to staging first, then production registry after approval (for curated registries).

**Reference implementation:** copy [docs/templates/github-actions/ion-library-ci.yml](templates/github-actions/ion-library-ci.yml) into `.github/workflows/` (see [templates/github-actions/README.md](templates/github-actions/README.md)). The **publish API** contract for step (4) is specified in [docs/PUBLISH_API.md](PUBLISH_API.md).

### 4. Quality and policy gates

Before a version is listed as non-yanked:

- **Build and tests** pass on agreed platforms.
- **Static analysis**: minimal bar from `ion check` (severity thresholds configurable per registry policy).
- **License**: SPDX identifier present and compatible with registry policy.
- **Metadata**: description, homepage, and dependency ranges reviewed (manually at first, automated checks later).

### 5. Governance

- **Namespaces** (`scope/name` vs flat names) and **name disputes**.
- **Security**: report channel, rapid yank for compromised releases, optional maintainer PGP or Sigstore signing of tarballs or index entries.

### 6. Suggested rollout phases

| Phase | Focus |
|-------|--------|
| **A — Curated bootstrap** | Manual or scripted index updates; few dozen high-value packages; validate the artifact contract end-to-end. |
| **B — Automation** | Reusable CI, staging registry, scripted publish from CI secrets. |
| **C — CLI publish** | `ion publish` (or a thin `ion-registry` CLI) talking to the publish API; API keys with scopes. |
| **D — Scale** | Search optimization, mirrors, download analytics, optional binaries. |

## What to implement in *this* repository

- **Client**: extend `IonRegistry` / resolver when the server adds fields or endpoints; optional `ion publish` subcommand once the API is stable.
- **Docs**: keep [README.md](../README.md) user-facing; keep this file updated as phases land.
- **Templates**: [docs/templates/](templates/README.md) (GitHub Actions for library authors); [PUBLISH_API.md](PUBLISH_API.md) for the registry upload contract.

## Summary

A **200+ package ecosystem** is not only “more JSON files”: it needs a **clear tarball contract**, a **registry backend** (storage + optional publish API), **automated CI** for package repos, and **policy gates** so the catalog stays trustworthy. The Ion CLI already implements the **consumer** side; the **producer** and **operations** layers above are the architectural prerequisite for that roadmap milestone.
